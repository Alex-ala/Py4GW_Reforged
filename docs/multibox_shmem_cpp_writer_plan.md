# Multi-account Shared Memory — move the writer to C++

> **Status: IMPLEMENTED and WORKING.** The C++ writer produces a byte-exact mirror of
> the Python layout (`sizeof(AllAccounts) == 940096`), Python reads it every frame,
> and the multibox shared memory functions. The one bug that blocked it — a wrong
> `len(Attribute)` constant (43 vs 46) that made the buffer too small — and the fix
> are recorded in `multibox_shmem_cpp_writer_POSTMORTEM.md`. Read that before
> touching the layout again.

## Intent

The multi-account shared-memory buffer (`Py4GW_Shared_Mem`, the `GLOBAL_CACHE.ShMem`
surface) is currently **created and written from Python**, every frame, via
`Py4GWSharedMemoryManager.update_callback`. That write path is the system's main
per-frame cost, and it is fought with an elaborate tiered/staged throttle so it fits
inside a frame.

This migration moves the **writer** into a new **C++ module** in
`Py4GW_Reforged_Native`, exactly the way Plane A (the system buffer,
`GW::shared_memory`) already works: **C++ creates and writes, Python attaches and
reads.** Nothing about the data model changes.

### Locked scope — only three things change

1. **C++ owns the push.** C++ creates `Py4GW_Shared_Mem` and writes it; Python goes
   read-only.
2. **Stable email is the identity.** Slot claim/find keys on the C++ account anchor
   (`PY4GW::System::Instance().GetAccountEmail()`, latches once and is immutable),
   not on the volatile HWND.
3. **Full push every frame, no throttle.** The Python `ThrottledTimer` tiers/stages
   are deleted; C++ writes every field every frame.

Everything else stays identical: same struct layout, same 64 slots, same slot logic,
same timeout liveness, same readers. The coordination regions (`Inbox` messaging,
`Intents` whiteboard, `HeroAIOptions`) are **generated by Python** and stay
Python-written — C++ has no source for them.

## Behavioral addition: zero on map-gate-fail

Today, when Python loses its email handle during a map load it cannot resolve its
slot and readers fall through to "assume zeroes" — accidental safety. With a stable
email the slot is *always* resolvable, so that accidental safety is gone: a reader
would find the slot and read whatever is in it. If C++ left last-frame data there
during a load, readers would consume **invalid game contexts**.

Therefore: **while a writer's map gates are not met, C++ zeroes the entity's payload
but keeps the slot latched** (Key / AccountEmail / entity flags / `IsSlotActive` /
`SlotNumber` / `LastUpdated`), so the slot stays findable-by-email and reads as
"owner present, data zeroed." (Plane A already does exactly this — it `ZeroMemory`s
its payload and returns when `is_map_ready` is false.)

## C++ module structure (matches the other modules)

New module `GW::multibox`, laid out like `GW::shared_memory`:

```
Py4GW_Reforged_Native/
  include/GW/multibox/manager.h    # byte-identical mirror structs + Manager class
  src/GW/multibox/manager.cpp      # lifecycle, seqlock, dispatch, per-frame fill
```

- Namespace `GW::multibox`. Class `Manager`, file-scope singleton + `RuntimeManager()`
  accessor (same shape as `GW::shared_memory::RuntimeManager()`).
- CMake needs **no edits** — `CMakeLists.txt` globs `src/**/*.cpp` with
  `CONFIGURE_DEPENDS`, so the new folder is picked up automatically.
- Wiring points in `src/Py4GW.cpp`, beside the Plane A manager:
  - `Py4GW_Initialize` → `multibox::RuntimeManager().CreateOrOpen(L"Py4GW_Shared_Mem")`
  - `UpdateLoopStep` → `multibox::RuntimeManager().Update()`
  - `Py4GW_Shutdown` → `multibox::RuntimeManager().Destroy()`

## The layout contract (the linchpin)

The C++ mirror structs must be **byte-identical** to the Python ctypes structs in
`Py4GWCoreLib/GlobalCache/shared_memory_src`:

- `#pragma pack(push, 1)` on all mirror structs.
- Type map: `c_uint`→`uint32_t`, `c_int`→`int32_t`, `c_uint64`→`uint64_t`,
  `c_float`→`float`, `c_bool`→`bool` (1 byte, MSVC), `c_wchar`→`wchar_t`
  (2 bytes on Windows — `static_assert(sizeof(wchar_t)==2)`). `Vec2f`/`Vec3f` =
  2/3 `float`.
- Constants: `MAX_PLAYERS 64`, `EMAIL_LEN 64`, `CHAR_LEN 64`, `AVAILABLE_CHARS 20`,
  `BUFFS 240`, `SKILLS 8`, `ATTRIBUTES 43` (`len(Attribute)`), `TITLES 48`,
  `QUESTS 150`, `INV_BAG_SLOTS 20`, `MISSION_BITMAP 25`, `SKILL_BITMAP 108`,
  `MAX_INTENTS 64`.
- Top-level `AllAccounts` order: `Keys[64]`, `AccountData[64]`, `Inbox[64]`,
  `HeroAIOptions[64]`, `Intents[64]`.

**Verify against live Python, not hand math.** Run once in the injected env and paste
the numbers into C++ `static_assert`s:

```python
import ctypes
from Py4GWCoreLib.GlobalCache.shared_memory_src.AllAccounts import AllAccounts
from Py4GWCoreLib.GlobalCache.shared_memory_src.AccountStruct import AccountStruct
print("AllAccounts", ctypes.sizeof(AllAccounts))
print("AccountStruct", ctypes.sizeof(AccountStruct))
```

If C++ `sizeof(AllAccounts)` != the Python number, the layout has drifted and Python
reads garbage. This assert is the migration's safety net.

## Lifecycle — create-or-open (differs from Plane A)

Plane A creates a *per-process* region (name = pid+hwnd) with plain
`CreateFileMappingW`. The multi-account buffer is **one shared region across all
instances** with a **fixed name** and **many writers**, so the manager must
create-or-open:

- `CreateFileMappingW(INVALID_HANDLE_VALUE, …, L"Py4GW_Shared_Mem")` returns the same
  object for every instance; `GetLastError()==ERROR_ALREADY_EXISTS` tells you whether
  you were the creator.
- **Only the creator zeroes** the whole region (matches Python's create-branch
  `ResetAllData`). Late joiners must not clobber existing slots.
- Same-session/same-machine only (given), so no cross-session naming concerns.

## Dispatch — keyed on the stable email

Each instance writes **only its own** account + heroes + pets. The claim/find logic
ports from `AllAccounts.py` unchanged in shape, but keyed on the stable anchor:

- Find own account slot by `AccountEmail == GetAccountEmail()` (+ entity type). Claim
  an empty/expired slot if none. C++ can make the claim atomic
  (`InterlockedCompareExchange` on a slot state word) since it owns the write.
- Heroes/pets recycle on map transitions exactly as today.
- Timeout liveness (`LastUpdated` + 5s) is retained as-is; C++ heartbeats the account
  slot **unconditionally** (even during a load) so a long map load can't false-expire
  it.

Python readers keep resolving slots locally by scanning the anchor already stamped in
each slot — a pure lookup now, no competing allocation, no C++→Python pointer dispatch
needed.

## Python read-only conversion (the delta)

In `Py4GWCoreLib/GlobalCache/`:

- **Writer out** — stop registering `update_callback` on `Phase.Data`;
  `SetPlayerData`/`SetHeroesData`/`SetPetData` and all `from_context` populate methods
  become dead (delete or leave dormant).
- **Attach-only** — remove the `create=True` branch in `SharedMemory.py.__init__`;
  a missing buffer means "no writer up yet," handle gracefully.
- **Throttle out** — delete the `ThrottledTimer` tiers, stage dicts, and throttle
  constants in `AccountStruct.py`, `AgentDataStruct.py`, `Globals.py`.
- **No write-on-miss** — `GetSlotByEmail` and friends must not `Submit*`; they become
  pure lookups returning `-1`/`None`.
- **Coordination stays** — `SendMessage`, `PostLock`/whiteboard, and `HeroAIOptions`
  writes remain Python.
- **Readers unchanged** — every getter keeps its signature; it just reads now.

## Sequencing

1. Land the C++ module header (byte-identical structs + Manager) and get the live
   `sizeof` numbers into `static_assert`s. — **structural, low-risk**
2. Implement `manager.cpp`: lifecycle, seqlock, dispatch, gate-zeroing, and port the
   per-struct field fill from the Python `from_context` methods to `GW::` accessors.
   — **the iterative, build-and-verify phase** (needs the live game + compiler)
3. Wire into `Py4GW.cpp` (create/update/destroy).
4. Flip Python to read-only.
5. Verify: attach a reader, confirm slots populate identically to the old Python
   writer; confirm zeroed payload during map loads; confirm no false-expiry across a
   long load.

Rollback: keep the Python writer switch-on-able until the C++ writer is validated at
parity — the two produce the same bytes, so you can A/B by toggling which one runs.
